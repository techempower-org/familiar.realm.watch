# familiar.realm.watch v0.3 — *the familiar writes back*

**Date:** 2026-04-26
**Status:** Spec, awaiting plan
**Supersedes:** the v0.3 milestone in `2026-04-23-familiar-realm-watch-design.md` (P40-only scope)

## One-line summary

Close the write-side gap (palace doesn't know what the LLM just synthesized), land the upstream PRs that already fix our biggest read-side issues, and consume the BM25 score that mempalace already returns.

## Why this scope (and why it changed)

The original v0.3 plan was hardware migration: P40 GPU + 14B model + palace-on-familiar. After the v0.2.1 release and first jp-realm-v0.1 baseline, the empirical signal said something different:

- The pipeline contribution (rerank + decay + compress) is real but small: **+3.33pp recall, never regresses**.
- 2 of 3 hard misses (q12 rlm, q13 GraphPalace) were **write-side gaps** — content synthesized in chat, never written to palace.
- The remaining hard miss (q08 kind=filter) is a **read-side problem we already have a filed-but-unmerged fix for** (mempalace #1005, BM25 top-up when vector underdelivers).

So v0.3's highest-leverage spend is *not* a smarter retriever. It's:
1. Writing back what was synthesized.
2. Following through on PRs that are already filed.
3. Consuming the bm25 score `/search` is already returning.

P40 hardware migration is real ops work but it's about *single-shot answer quality*, not *what palace knows*. It moves to a separate v0.3.5 ops track.

## Goals

- **G1.** Close the synthesized-but-unwritten gap. After v0.3, content discussed in chat that survives a quality gate ends up as palace drawers without explicit user request.
- **G2.** Merge the two highest-leverage of JP's open mempalace PRs (#1173 HNSW quarantine, #1005 BM25 top-up). After merge, familiar's q08-class misses go away and HNSW outages stop requiring the auto-repair workaround.
- **G3.** Use the bm25 score familiar already gets back from `/search`. Today's rerank ignores it.
- **G4.** Demonstrate the eval-corpus loop as the v0.3 dev workflow. Every shipped change re-runs jp-realm-v0.1 (and any new corpus); commits include the recall delta.

## Non-goals

- Building rlm into familiar's chat path. Multipass-side `RlmAdapter` is fine if it earns its way in via eval data, but no productionization commitment.
- Building hermes-agent or mempalace-viz. Both are real but separate efforts; v0.4 candidates.
- Migrating off ChromaDB. Postgres backend is upstream's #665 story, not ours.
- Authoring a new model story (P40 + 14B). That's v0.3.5 ops; tracked separately.
- New corpus authoring. jp-realm-v0.1 is enough to measure v0.3 work; expand only if specific gaps emerge.

## Architecture

Five subsystems, four of them small. Reflect is the only piece of greenfield software.

### A — Reflect loop *(greenfield, ~1 week)*

Per-turn pipeline that runs after the assistant's response is sent and the user's next message is received (so we can use the next turn as evidence the prior synthesis was accepted). Extracts novel facts from the assistant's prior turn, deduplicates against palace, writes survivors as drawers.

```
[turn N: assistant response]
[turn N+1: user message]
        │
        ▼
  ReflectExtractor
        │  yields: candidate facts (1 sentence each, with source span)
        ▼
   QualityGate
        │  drops: "I don't know"-shaped, contradictory, hedged, refused
        ▼
    DedupCheck
        │  palace.search(fact_summary) → if top sim > 0.85, skip
        ▼
   palace.writeMemory(content, wing="reflect", room=<inferred>)
```

Constraints:

- **Idempotent.** Running on the same turn twice writes nothing the second time.
- **Off the chat hot path.** Reflect runs after the response is sent. Failures don't affect the visible UX.
- **Quality-gated.** Better to write nothing than to write hallucinations. Conservative thresholds; tune over time.
- **Wing-scoped.** All reflect-written drawers go to a `reflect` wing (or per-session-derived wing) so they're attributable and removable in bulk.
- **Observable.** Every reflect decision (extracted, gated, deduped, written) emits a journal line with reasons, mirroring v0.2's Trace pattern.

#### Components

- `src/reflect/extractor.ts` — turn-text → candidate facts. Initial implementation: prompt-based extraction via the same InferenceRouter familiar already has. Future: structured extraction with a smaller dedicated model.
- `src/reflect/gate.ts` — quality filter. Pure metadata math + a small set of pattern checks initially.
- `src/reflect/dedup.ts` — palace.search-based dedup. Returns `seen | novel`.
- `src/reflect/writer.ts` — orchestrates the above; calls `palace.writeMemory`. One public method.
- `src/reflect/types.ts` — `ReflectCandidate`, `ReflectDecision`.
- `src/sessions.ts` — wire the reflect call after each assistant turn (or batched per N turns, decision below).

#### Open design questions (resolve in plan)

1. **Per-turn vs per-session.** Per-turn means every accepted answer triggers extraction (high frequency, high cost, fast feedback). Per-session means extraction runs on `Stop` hook over the whole transcript (cheaper, lower fidelity per fact, leverages DiaryBuffer).
   - **Recommended:** per-session for v0.3, per-turn behind a flag for v0.4. Reuses DiaryBuffer's flush rhythm; simpler quality gate; smaller batch size to debug.
2. **Extraction prompt.** Initial proposal: "Extract up to N novel factual claims from the assistant's previous turn. Discard hedged or refused content. Return as a JSON array of `{fact: string, source_span: [start, end]}`."
3. **Dedup threshold.** Start at cosine ≥ 0.85 against palace.search top-1; tune via the corpus.
4. **Wing assignment.** Heuristic per-fact based on subject keywords, or always `reflect/<session_id>`. Heuristic is harder but discoverable in the long run.

### C — Consume bm25 in familiar's rerank *(surgical, ~1 day)*

`palace-daemon`'s `/search` already returns `bm25` per drawer (the mempalace fork surfaces it; familiar's `PalaceDrawer` type already carries it). Familiar's `src/retrieval/rerank.ts` weights `cosine + wing_match × 1.4 + recency`. **It ignores `bm25`.** Adding a `bm25` term gets us q08 coverage *before* #1005 merges upstream.

Concrete change:

```ts
// src/retrieval/rerank.ts (sketch)
const blendedScore =
  drawer.cosine * 0.6 +
  (drawer.bm25 ?? 0) * 0.4;            // ← new
const wingBoost = wingMatch ? 1.4 : 1.0;
const recencyBoost = recencyMultiplier(drawer.date);
return blendedScore * wingBoost * recencyBoost;
```

Weights tunable; start at 0.6/0.4 and re-run jp-realm-v0.1 with three weight pairs (0.5/0.5, 0.6/0.4, 0.7/0.3) to pick. Document the chosen weight in CHANGELOG.

### D — File the trailing-punctuation fix upstream *(small contribution, ~half day)*

Today the fix is in familiar's `src/palace-client.ts`. Every nomic-embed-text v1.5 consumer faces the same issue. Upstream PR target: `mempalace.searcher` query-side normalization. Strip trailing `[?!.,;:]` before embedding; document the why in the docstring; one test case from jp-realm-v0.1 (`What is GraphPalace and how does its pheromone model work?` → returns the right drawer at top-1).

After merge: revert the client-side normalization in familiar in favor of the canonical upstream fix.

## Data flow (v0.3 chat turn — additive over v0.2)

```
[v0.2 path unchanged: retrieve → ground → answer → cite]
                                                      │
                                          [response sent to user]
                                                      │
                                          ┌───────────┴───────────┐
                                          │                       │
                              [user reads it; types next message] │
                                          │                       │
                                  (next turn arrives)              │
                                          │                       │
                                          ▼                       │
                                  ReflectWriter.review(prevTurn)  ← OFF HOT PATH
                                          │
                              extract → gate → dedup → write
                                          │
                                  palace.writeMemory(...)
                                          │
                                  journal: reflect.<decision>
```

## Error handling

Per familiar's existing principle (themed-voice graceful degradation): every reflect failure mode degrades silently to "do nothing." The chat path never blocks on reflect. If extraction fails, we log and skip. If palace is unreachable (rare on disks), we drop the reflect attempt rather than queueing it — DiaryBuffer already provides a heavier-weight session-end safety net.

## Testing approach

- **Unit:** extractor returns expected candidates on canned turns; gate filters known refusal patterns; dedup correctly returns `seen` for paraphrases above threshold and `novel` below.
- **Integration:** end-to-end reflect on a recorded conversation (against a test palace), verify expected drawers land, dedup correctly skips re-runs.
- **Eval:** after each subsystem ships, re-run jp-realm-v0.1 corpus through `/api/familiar/eval` and commit the baseline delta. New "hard-questions" corpus authored if reflect surfaces queries jp-realm-v0.1 doesn't cover.

## Order of operations

1. **C (consume bm25) first.** Smallest, fully under our control, immediately measurable on the corpus. Establishes baseline-delta workflow for v0.3.
2. **D (punctuation upstream PR) in parallel** — file early so review can run concurrently.
3. **B (drive #1173 and #1005 to merge)** — review-cycle work; threaded through the rest.
4. **A (reflect loop)** — main software effort. Lands after B has merged enough that read-side is settled.
5. **Re-run baseline** at each milestone; commit JSON to `multipass-structural-memory-eval/baselines/`.

## Risks

- **#1173 / #1005 review timeline is not under our control.** If upstream review stalls, item B becomes "follow up" rather than "land before A". Doesn't block reflect; means familiar consumes the BM25 top-up via fork pin in the interim.
- **Reflect quality gate is hard.** Extraction will surface hallucinations. The gate is the project; cheap to underwrite (write nothing) but expensive to over-write (palace pollution). Conservative thresholds first; tune via the corpus.
- **bm25 weight tuning is corpus-dependent.** What helps q08 might hurt currently-strong queries. Three-point sweep on the corpus before committing weights to main.
- **Reflect's per-session vs per-turn decision affects DiaryBuffer wiring.** Decided in plan; default per-session for v0.3.
